## 🎯 问题背景

### 初始问题
在执行 TTS 生成测试时，服务端在多次测试后出现内存占用过高的现象。

### 环境信息
- **操作系统**: macOS
- **设备**: MPS (Metal Performance Shaders)
- **总内存**: 16 GB
- **交换空间**: 6 GB
- **MPS 显存上限**: ~18.13 GB

### 核心问题
1. **MPS 显存耗尽**: `RuntimeError: MPS backend out of memory (MPS allocated: 18.12 GiB, max allowed: 18.13 GiB)`
2. **内存峰值递增**: 每次测试内存峰值都在增加
3. **交换空间占用高**: 达到 96%+，严重影响性能

---

## 🧠 内存优化策略

### 1. 显式内存清理
- **策略**: 在每次生成后强制执行垃圾回收和显存清理
- **实现**: 
  - 多次调用 `gc.collect()`
  - MPS 设备调用 `torch.mps.empty_cache()`
  - 添加延迟确保清理完成

### 2. 临时对象删除
- **策略**: 显式删除不再需要的临时对象
- **实现**:
  - 删除 `clone_prompt`、`wavs` 等临时变量
  - 在生成后立即删除音频数据

### 3. 异步处理
- **策略**: 使用线程池执行同步生成函数，避免事件循环阻塞
- **实现**:
  - `loop.run_in_executor()` 包装同步函数
  - 移除信号量限制，简化并发逻辑

## 🔧 实施方案

### 代码修改

#### 1. 增强内存清理函数
**文件**: `src/qwen3_tts_api/services/tts.py`

```python
def _cleanup_generation_memory():
    """清理生成过程中的临时内存"""
    import gc
    import time
    
    # 多次垃圾回收
    gc.collect()
    time.sleep(0.1)
    gc.collect()
    time.sleep(0.1)
    gc.collect()
    time.sleep(0.1)
    
    device = get_device()
    if device == "cuda":
        torch.cuda.empty_cache()
        torch.cuda.synchronize()
        torch.cuda.empty_cache()
        torch.cuda.synchronize()
    elif device == "mps":
        # 激进的 MPS 显存清理
        torch.mps.empty_cache()
        time.sleep(0.2)
        torch.mps.empty_cache()
        time.sleep(0.2)
        torch.mps.empty_cache()
```

## 🔍 问题诊断

### 根本原因分析

#### 1. 内存峰值问题
- **现象**: 内存峰值仍然存在（最高93.1%），但有所改善
- **原因**:
  - 模型加载和推理需要大量内存
  - MPS 显存管理机制限制
  - 临时对象和中间计算结果占用内存

#### 2. 交换空间占用增加
- **现象**: 交换空间逐次增加（75.0% → 79.2% → 85.4%）
- **原因**:
  - 物理内存不足（16GB）
  - 模型和数据占用大量内存
  - 系统频繁使用交换空间

#### 3. 内存释放机制有效
- **现象**: 每次测试后内存都能有效释放到60%左右
- **原因**:
  - 增强的内存清理函数工作正常
  - 临时对象删除机制有效
  - 垃圾回收和显存清理策略得当
